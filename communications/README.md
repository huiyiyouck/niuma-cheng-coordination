# communications — 项目间协作主纲

本目录承接 niuma-cheng 生态内**多组项目之间**的协作主纲文档。每份文档是**一组项目之间所有跨项目协作的总账与管理入口**。生态扩展时会有多份文档共存，每份对应一组参与项目。

## 模板定位

一份 `communications` 文档 = 一组项目之间协作的**主纲**：

- **双向对等**：不预设谁是需求方、谁是承接方。任一参与方都可向其他方提需求。
- **不等于技术调用方向**：HTTP/依赖的调用方向是事实，但需求是双向的，二者不绑定。
- **一份看全**：读这一份就掌握「这组项目之间有哪些协作诉求 + 每条做到哪了 + 正在联调什么」。
- **管两层**（见下）。契约细节在 `../contracts/`，全局状态在 `../STATUS.md`，本目录是协作的叙事/管理主线。

## 每份文档管两层

| 层 | 作用 | 内容 |
|----|------|------|
| **A 需求沟通** | 项目之间提出的协作需求（任一方可提） | 需求台账：需求 id / 提出方 → 面向方 / 内容 / 对应契约 / 状态 |
| **B 联调沟通** | 每条需求确立**之后**的持续往来 | 接口对接、字段对齐、调试问题、版本跟进（倒序时间线，条目标注所属需求 id） |

涉及接口/字段的需求，其契约以 [../contracts/](../contracts/) 为单一真源；本文件只引用，不复制契约正文。

## 需求流转与角色权限

- **提报**：任一项目的**任一角色**都可提报需求。
- **承接 / 拒绝**：仅**目标项目的 PM 或 Architect**；其他角色不得代为承接。
- **承接后**：目标项目把需求转为本项目内部标准迭代开发，本目录只登记「转入 {项目} vX.Y」并链接回，不复制迭代细节；开发完成回到 B 区联调。
- **状态机**：已提报 → 评估中 → 已承接（/已拒绝）→ 开发中（转入迭代）→ 联调中 → 已关闭。

完整规则见各项目 `docs/baseline/cross-project-collaboration.md` §跨项目需求流转。

## 命名规则

- 两个项目：`{project-a}__{project-b}.md`，如 `xiaobao__ai.md`
- 三个及以上：`{project-a}__{project-b}__{project-c}.md`
- 项目 id 用 [../PROJECTS.md](../PROJECTS.md) 中的稳定短名
- 文件名只表达**参与方**，不表达临时议题；议题写进文档内章节
